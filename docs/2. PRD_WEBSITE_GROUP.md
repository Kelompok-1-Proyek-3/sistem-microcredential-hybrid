## 1. Executive Summary

Website Group bertanggung jawab untuk **core platform** Microcredential IMPC. Modul eLearning adalah **critical path** yang mengintegrasikan seluruh workflow (enrollment, content delivery, assessment, sesi hybrid offline, scan check-in, certificate generation, external verification). eCommerce mendukung B2C catalog exposure dan checkout experience.

**Deliverables**:
- eLearning module enhancement (kustomasi signifikan)
- Hybrid learning mode management (Online/Offline/Hybrid)
- Offline session scheduling + e-ticket generation
- Admin scan check-in + exam gating by attendance
- eCommerce → eLearning integration (kustomasi minor)
- REST API untuk external systems

#### User Story 5B: Exam Gating by Offline Attendance (Hybrid)

**Given**: Course mode adalah `HYBRID`  
**When**: Student mencoba membuka Ujian Akhir  
**Then**: Sistem hanya membuka ujian jika status attendance sesi offline = `HADIR`

**Acceptance Criteria**:
- [ ] Jika `ONLINE`: ujian terbuka tanpa cek attendance
- [ ] Jika `HYBRID`: cek `elearning.session_attendance.attendance_status`
- [ ] Jika belum hadir, tampilkan pesan gating + instruksi sesi offline
- [ ] Jika sudah hadir, akses ujian dibuka otomatis
- [ ] Gating check bersifat idempotent dan tahan retry
- [ ] Semua keputusan gating tercatat di log

**Validation Logic**:
```python
def can_open_final_exam(self, enrollment):
  if enrollment.course_id.learning_mode == 'ONLINE':
    return True
  if enrollment.course_id.learning_mode == 'HYBRID':
    attendance = self.env['elearning.session_attendance'].search([
      ('enrollment_id', '=', enrollment.id),
      ('attendance_status', '=', 'HADIR')
    ], limit=1)
    return bool(attendance)
  return True
```

**Database Schema**:
```
elearning.module_completion:
  - id (primary)
  - enrollment_id (FK to course_enrollment)
  - module_id (FK to elearning.module)
  - completion_date (datetime)
  - time_spent_minutes (int)
  - quiz_score (float, nullable)
  - quiz_attempts (int, default 1)
  - created_at (datetime)
```

**Tracking Logic**:
- Client-side: JavaScript event listeners for module completion
- Server-side: API endpoint `/api/v1/elearning/track-completion` (POST)
- Payload validation: Student must be enrolled, module must exist, etc.

#### User Story 6: Learning Completion Rules

**Given**: Student progress tracking is enabled  
**When**: Student reaches completion_percentage=100% AND quiz_passed=true AND (if hybrid) attendance_status='HADIR'  
**Then**: Course marked as completed, eligible for certificate

**Acceptance Criteria**:
- [ ] Completion trigger: Auto-detect when conditions met
- [ ] Blocking rule: If progres<100% OR quiz_passed=false OR (hybrid and not checked-in), block certificate generation
- [ ] Completion date: Record exact timestamp of completion
- [ ] Status change: `student_progress.status = 'COMPLETED'`
- [ ] Certificate eligibility: Auto-flag for certificate generation workflow

**Validation Logic** (Odoo constraint):
```python
@api.constrains('completion_percentage', 'quiz_passed', 'attendance_status')
def _check_completion_eligibility(self):
    for record in self:
    if record.completion_percentage == 100.0 and not record.quiz_passed:
            raise ValidationError("Student harus lulus kuis sebelum completion")
    if record.course_id.learning_mode == 'HYBRID' and record.completion_percentage == 100.0 and record.attendance_status != 'HADIR':
      raise ValidationError("Course hybrid mensyaratkan check-in offline sebelum completion")
```

**Then**: System generates certificate with QR Code & Hash ID

  - Student name, course name, completion date
  - QR code (2x2 inch @ 96 DPI)
  - Hash ID (hex format, 64 chars)
  - Certificate number (sequential: CERT-YYYY-XXXXX)
  - Digital signature (if applicable)
- [ ] Certificate storage: Saved to student profile + accessible via portal
- [ ] Database record: `elearning.certificate` table with all metadata
- [ ] Versioning: Support certificate template versioning (different designs per course/year)

**Database Schema**:
```
elearning.certificate:

  - issued_date (datetime)
  - expiry_date (date, nullable)

  - template_id (FK to certificate_template)
```

### 5.3 Security Considerations

- **Redeem Code**: Store hashed (not plain) + use secure random generation (not sequential)
- **Certificate Hash**: 256-bit SHA-256 HMAC, verified on every access
- **API Rate Limiting**: 100 req/min per IP for certificate verification (prevent brute force)
- **CSRF Protection**: All POST endpoints protected by Odoo CSRF tokens
- **SQL Injection**: Use parameterized ORM queries (no string interpolation)
- **XSS Protection**: HTML templates escaped by default in Odoo
- **Audit Logging**: Log all enrollment, redeem, and certificate operations
- **Data Privacy**: Student PII encrypted at rest (if required by regulation)
