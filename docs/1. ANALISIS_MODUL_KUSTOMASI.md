# Analisis Modul Odoo: Kebutuhan Kustomasi vs Standar
**Referensi**: Proses Bisnis Microcredential Platform IMPC (4 Skenario BPMN) + Update Hybrid/Check-in (Skenario 1 & 3)
**Platform**: Odoo 19.0 (Community atau Enterprise) — direvisi berdasarkan Brief IMPC v2.0 (Revisi Faktual)

---

## Executive Summary

> **Prinsip Analisis (Odoo 19)**: Maksimalkan penggunaan fitur native Odoo 19 terlebih dahulu. Kustomasi hanya dilakukan jika fitur native benar-benar tidak mencukupi kebutuhan bisnis. Beberapa item yang sebelumnya diasumsikan perlu custom dev ternyata sudah tersedia native di Odoo 19.

Berdasarkan analisis BPMN 4 skenario dengan mengoptimalkan fitur native Odoo 19, sistem Microcredential Platform IMPC membutuhkan **1 modul dengan kustomasi signifikan** (eLearning — hanya 4 item spesifik), **5 area kustomasi minor**, dan **9 modul/fitur standar tanpa kustomasi**.

**Resiko Integrasi**: Terkonsentrasi pada 4 item di eLearning: Redeem Code engine, Exam Gating hybrid, QR/Hash verifikasi publik, dan External Verification API. Koordinasi ketat diperlukan antara tim Website, Accounting, dan Marketing (Events).

### SKENARIO 1: Manajemen Konten, Tipe Pembelajaran, & Setup B2B

| Proses Bisnis | Modul Odoo Relevan | Status | Catatan |

## 4. Daftar Modul dengan Kustomasi Signifikan / Minor

### 🔴 KUSTOMASI SIGNIFIKAN (Custom Development — Tidak Tersedia Native di Odoo 19):

> Hanya 4 item berikut yang benar-benar memerlukan custom development di Odoo 19.

1. **eLearning** (Website Group)
   - **Redeem voucher validation engine** — bulk code generation, claim validation, marking `isClaimed` (tidak ada native Redeem Code system di Odoo 19)
   - **Exam gating berdasarkan Events attendance (mode Hybrid)** — eLearning tidak punya native linking ke status hadir Events; perlu custom gating logic
   - **QR Code + Hash ID pada sertifikat untuk verifikasi publik** — sertifikat native tidak menyertakan hash unik untuk external lookup
   - **External certificate verification API endpoint** — halaman verifikasi publik standalone TIDAK tersedia native di Odoo 19; perlu custom development yang dianggarkan terpisah

---

### 🟡 KUSTOMASI MINOR (Konfigurasi / Minor Code — 1-2 User Stories per Item):

1. **eLearning** (Website Group)
   - Custom field `learning_mode` (Online/Offline/Hybrid) pada form kursus
   - Custom field `bobot_sks` untuk penandaan internal RPL (hanya referensi internal — integrasi ke PDDikti/SISTER butuh scope terpisah)
   - Sinkronisasi status hadir Events → eLearning untuk trigger exam gating

2. **Sales** (Sales Group)
   - B2B contract template enhancement (tambahan klausul learning mode & jadwal hybrid)

3. **eCommerce** (Website Group)
   - Product catalog linking ke eLearning (UX integration)

4. **Accounting** (Accounting Group)
   - Install + konfigurasi modul third-party Midtrans/Xendit dari Odoo Apps Store
   - Redeem code transaction reconciliation logic

5. **HR / Skills** (HR Group)
   - Progress reporting view (Pivot/Graph) dari data eLearning per batch
   - Competency mapping dari certificate completion ke skill matrix (**hanya untuk hr.employee** — bukan peserta umum / Portal User)

6. **Events** (Marketing Group)
   - Linking Events registration ke eLearning enrollment untuk sinkronisasi status hadir ke exam gating

---

### 🟢 STANDAR — Native Odoo 19 (0 Kustomasi):

1. **eLearning** (Website Group)
   - Certificate PDF generation + pengiriman otomatis via email saat lulus (aktifkan: eLearning → Configuration → Settings → Certifications)
   - Progress tracking peserta (eLearning → Attendees)
   - Content management: Video, Dokumen, Kuis, Ujian Akhir
   - Enrollment otomatis on payment (Enroll Policy: On Payment)
   - Badge di profil portal peserta

2. **Events** (Marketing Group)
   - Manajemen jadwal, venue, kapasitas, dan tiket sesi tatap muka
   - QR scan check-in via Odoo Mobile App — tersedia di **Community & Enterprise** (bukan eksklusif Enterprise)
   - Konfirmasi registrasi peserta (e-ticket native)

3. **eCommerce / Website** (Website Group)
   - Katalog kursus online & checkout B2C

4. **CRM** (Sales Group)
5. **Contacts / res.partner** (Sales Group) — peserta umum disimpan di sini sebagai Portal User, **bukan** sebagai Employee
6. **Invoicing** (Accounting Group) — konfigurasi standar + Indonesian Localization PPN 11%
7. **Surveys** — terinstall otomatis sebagai dependensi eLearning; assessment/certifications dikelola dari eLearning, **tidak dikonfigurasi terpisah**

## 5. Boundary & Scope Definition per Group

### 🚫 BOUNDARY RULES untuk Menghindari Konflik

#### Website Group (Core eLearning)
- **Owns**: eLearning module, learning mode schema (custom field), exam gating logic (hybrid), Redeem Code engine, certificate hash/QR schema, enrollment logic, external verification API
- **Uses native**: Certificate PDF generation, progress tracking, content management, enrollment on payment — **jangan re-implement**
- **Must NOT**: Touch Accounting invoice records directly (use API/hooks); create HR employee records (peserta umum → Contacts/Portal User)
- **Interface**: Event-driven sync dari Events attendance untuk exam gating; Webhook untuk payment validation dari Accounting

#### Accounting Group
- **Owns**: Payment processing (via third-party module), invoice generation, financial reconciliation, konfigurasi PPN 11%
- **Must NOT**: Modify eLearning course prices (via config only); create/modify Redeem codes (use API provided by Website)
- **Interface**: Payment confirmation webhook ke Website (trigger enrollment); Reconciliation untuk Redeem Code transactions

#### Sales Group
- **Owns**: B2B contract management, pricing negotiation
- **Must NOT**: Modify course content (reference only); create Redeem codes (request dari Website API)
- **Interface**: Contract data → Website modul via foreign key; Redeem code generation trigger

#### HR Group
- **Owns**: Employee database (`hr.employee`), skills management (employee only — **bukan untuk peserta umum**), progress reporting view
- **Must NOT**: Modify course content atau certificate data; create enrollment records (read-only access)
- **Catatan Odoo 19**: Skill management (`hr_skills`) **hanya berlaku untuk hr.employee**. Peserta umum dikelola di Contacts (res.partner) sebagai Portal User — gunakan Tags atau custom field via Studio untuk tracking kompetensi peserta.
- **Interface**: Read-only eLearning progress via Pivot/Graph View atau custom reporting API

#### Marketing Group (Events)
- **Owns**: Event management (jadwal, venue, tiket, kapasitas), QR scan check-in via Odoo Mobile App, absensi sesi tatap muka
- **Uses native**: Events + Odoo Mobile App untuk check-in (tersedia di Community & Enterprise)
- **Must NOT**: Touch eLearning enrollment atau exam gating logic
- **Interface**: Attendance status Events → Website Group untuk sync ke exam gating; campaign linking ke eCommerce catalog (config only)

## 7. Risk Matrix

| Risk | Severity | Mitigation |
|---|---|---|
| Conflict pada eLearning schema (Redeem Code / Gating) | HIGH | Define schema lock-down; Website group owns eLearning schema |
| Payment integration failure (third-party module) | HIGH | Verifikasi kompatibilitas modul Midtrans/Xendit dengan Odoo v19 sebelum produksi; siapkan fallback manual |
| Redeem code data inconsistency | HIGH | Implement distributed lock mechanism; atomic DB transaction |
| Gating ujian tidak sinkron dengan Events attendance | HIGH | Event-driven attendance update + idempotent gate check |
| Certificate verification spoofing (public API) | HIGH | Implement HMAC signing + timestamp validation pada Hash ID |
| Kompatibilitas modul third-party Midtrans/Xendit dengan Odoo 19 | MEDIUM | Test di staging environment sebelum production; siapkan anggaran alternatif jika modul belum support v19 |
| Real-time dashboard performance | MEDIUM | Gunakan Pivot View native + cache layer untuk aggregasi custom |
| Cross-module data sync delays (Events → eLearning) | MEDIUM | Use Odoo bus (event-driven) untuk attendance sync |
| Skills mapping salah target (employee vs peserta) | LOW | Enforce boundary: hr_skills HANYA untuk hr.employee; peserta → Contacts Tags |
