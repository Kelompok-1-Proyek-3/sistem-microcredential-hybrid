# Analisis Modul Odoo: Kebutuhan Kustomasi vs Standar
**Referensi**: Proses Bisnis Microcredential Platform IMPC (4 Skenario BPMN) + Update Hybrid/Check-in (Skenario 1 & 3)

---

## Executive Summary

Berdasarkan analisis BPMN 4 skenario + update hybrid/check-in, sistem Microcredential Platform IMPC membutuhkan **3 modul dengan kustomasi signifikan**, **3 modul dengan kustomasi minor**, dan **4 modul standar tanpa kustomasi**.

**Resiko Integrasi**: Tinggi pada modul eLearning (core) dan Payment Integration. Koordinasi ketat diperlukan antara tim Website dan Accounting.

---

## 2. Mapping Modul Odoo → Tim / Group

### Sales Group
**Modul**: Sales, CRM, Contacts

| Modul | Kebutuhan | Scope Kustomasi |
|---|---|---|
| **Sales** | B2B Contract Management | MINOR: Perlu template contract khusus untuk Microcredential B2B (SLA khusus) |
| **CRM** | Lead management & HR Perusahaan relationship | STANDAR: Tidak ada kustomasi, gunakan fitur native |
| **Contacts** | Partner/Customer database | STANDAR: Tidak ada kustomasi, native record organization |

**Kolaborasi Kritis**: Dengan Accounting (untuk contract financial terms) dan Website (untuk B2B contract linking)

---

### Website Group
**Modul**: eLearning, eCommerce

| Modul | Kebutuhan | Scope Kustomasi |
|---|---|---|
| **eLearning** | **CORE PLATFORM** | **KUSTOMASI SIGNIFIKAN**: Redeem voucher system, mode hybrid/offline, e-ticket & scan check-in, exam gating, certificate generation, enrollment tracking, progress dashboard |
| **eCommerce** | Katalog produk kursus, checkout B2C | MINOR: Product catalog linking ke eLearning, UX integration |

**Kolaborasi Kritis**: Dengan Accounting (payment), Marketing (events/campaigns), Sales (B2B linking)

**Resiko Kolaborasi**: Tertinggi - eLearning adalah core. Pastikan interface dengan module lain clearly defined.

---

### Accounting Group
**Modul**: Accounting, Invoicing

| Modul | Kebutuhan | Scope Kustomasi |
|---|---|---|
| **Accounting** | Payment processing, invoice generation | KUSTOMASI MINOR: Payment gateway integration hooks, reconciliation untuk Redeem Code transactions |
| **Invoicing** | Invoice untuk B2C & B2B | STANDAR: Native invoicing, cukup konfigurasi |

**Kolaborasi Kritis**: Dengan Website (payment validation), Sales (B2B invoice generation)

---

### HR Group
**Modul**: HR, Skills

| Modul | Kebutuhan | Scope Kustomasi |
|---|---|---|
| **HR** | Employee learning tracking, attendance hybrid & dashboard | KUSTOMASI MINOR: Real-time progress dashboard + attendance check-in integration dengan eLearning |
| **Skills** | Skill mapping dari course completion | KUSTOMASI: Mapping competency dari course certificate ke skill matrix |

**Kolaborasi Kritis**: Dengan Website (eLearning data sync), Accounting (training budget allocation - opsional)

---

### Marketing Group
**Modul**: Events, Surveys

| Modul | Kebutuhan | Scope Kustomasi |
|---|---|---|
| **Events** | Dukungan jadwal sesi tatap muka hybrid (opsional) | MINOR: Konfigurasi sinkron jadwal dari course hybrid (tanpa custom code inti) |
| **Surveys** | Post-course feedback & satisfaction survey | STANDAR: Native survey features cukup |

**Kolaborasi Kritis**: Dengan Website (campaign linking ke catalog), Sales (event lead generation - minimal)

## 5. Boundary & Scope Definition per Group

### 🚫 BOUNDARY RULES untuk Menghindari Konflik

#### Website Group (Core)
- **Owns**: eLearning module, learning mode schema, offline session schedule, e-ticket/check-in, exam gating, certificate schema, enrollment logic, progress tracking
- **Must NOT**: Touch Accounting invoice records directly (use API/hooks)
- **Must NOT**: Create HR employee records (use established Contacts)
- **Interface**: REST API untuk progress sync ke HR; Webhook untuk payment validation dari Accounting

#### Accounting Group
- **Owns**: Payment processing, invoice generation, financial reconciliation
- **Must NOT**: Modify eLearning course prices (via config only)
- **Must NOT**: Create/modify Redeem codes (use API provided by Website)
- **Interface**: Payment webhook ke Website; Reconciliation API ke Website

#### Sales Group
- **Owns**: B2B contract management, pricing negotiation
- **Must NOT**: Modify course content (reference only)
- **Must NOT**: Create Redeem codes (request dari Website API)
- **Interface**: Contract data → Website modul via foreign key; Redeem code generation trigger

#### HR Group
- **Owns**: Employee database, skills management, attendance monitoring dashboard (read model)
- **Must NOT**: Modify course content atau certificate data
- **Must NOT**: Create enrollment records (read-only access)
- **Interface**: Read-only eLearning progress via custom view/API

#### Marketing Group
- **Owns**: Event management, survey templates
- **Must NOT**: Touch eLearning atau enrollment logic
- **Interface**: Campaign linking ke Website catalog + sinkron jadwal sesi hybrid (config only)

## 7. Risk Matrix

| Risk | Severity | Mitigation |
|---|---|---|
| Conflict pada eLearning schema | HIGH | Define schema lock-down; Website group owns |
| Payment integration failure | HIGH | Implement retry logic; Accounting owns reconciliation |
| Redeem code data inconsistency | HIGH | Implement distributed lock mechanism |
| Scan check-in tiket tidak valid / dipalsukan | HIGH | Signed QR token + one-time scan + expiry window |
| Gating ujian tidak sinkron dengan attendance | HIGH | Event-driven attendance update + idempotent gate check |
| Real-time dashboard performance | MEDIUM | Use materialized view / cache layer |
| Certificate verification spoofing | MEDIUM | Implement HMAC signing + timestamp validation |
| Cross-module data sync delays | MEDIUM | Use event-driven architecture (Odoo bus) |
