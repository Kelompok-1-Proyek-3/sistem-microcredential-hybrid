**Deliverables**:
- Payment gateway integration (kustomasi signifikan)
- Redeem code transaction mapping (kustomasi minor)

- Invoice generation & reconciliation (standar + config)

#### User Story 2: Online Course Payment Processing

**Given**: Customer initiates B2C course purchase with payment  
**When**: Customer submits payment details on eCommerce checkout  
**Then**: Payment gateway processes transaction

**Acceptance Criteria**:
- [ ] Payment initiation: System submits payment request to gateway API
- [ ] Payment details: Transferred securely (no PCI DSS violation)
- [ ] Payment reference: Unique transaction ID stored in Odoo
- [ ] Amount validation: Verify course price matches submitted amount
- [ ] Customer data: Customer info matched (name, email, phone optional)
- [ ] Payment state tracking: record payment status (pending, processing, paid, failed, cancelled)
- [ ] Timeout handling: If payment response not received after 30s, mark as "processing" & retry
- [ ] Duplicate detection: Prevent double-charging via idempotency keys

**Database Schema**:
```
payment.transaction (extended):
  - id (primary)
  - sale_order_id (FK to sale.order)
  - gateway_id (FK to payment.gateway)
  - reference (varchar, unique, indexed)  # Transaction ID from gateway
  - amount (float)
  - currency (char, e.g., 'IDR', 'USD')
  - state (selection: 'DRAFT', 'PENDING', 'AUTHORIZED', 'DONE', 'CANCELLED', 'ERROR')
  - failure_reason (text, nullable)
  - customer_id (FK to res.partner)
  - customer_email (varchar)
  - customer_phone (varchar, nullable)
  - created_at (datetime)
  - updated_at (datetime)
  - confirmed_at (datetime, nullable)
  - gateway_response (json, for debugging)
  - idempotency_key (varchar, unique)
```

**Payment Flow**:
```
1. Customer clicks "Pay Now" on eCommerce checkout
2. System creates payment.transaction record (state='DRAFT')
3. System calls gateway API: POST /charge
   - Payload: amount, currency, customer_email, sale_order_id, course_info
4. Gateway returns: transaction_id, payment_url (for redirect/modal)
5. Customer completes payment on gateway (Stripe Checkout, etc.)
6. Gateway redirects back to Odoo: /payment/confirm?reference=<tx_id>
7. System verifies transaction with gateway API
8. System updates payment_transaction.state based on response
9. If state='DONE' (paid), trigger enrollment webhook
```

#### User Story 3: Payment Webhook Reception & Processing

**Given**: Payment completed at gateway  
**When**: Gateway sends webhook to Odoo  
**Then**: System processes payment confirmation

**Acceptance Criteria**:
- [ ] Webhook endpoint: Active at `/payment/webhook/<gateway_code>`
- [ ] Authentication: Verify webhook signature (HMAC-SHA256 or gateway-specific)
- [ ] Idempotency: Same webhook payload processed only once (use reference key)
- [ ] Status mapping: Map gateway status → Odoo payment_state
  - Gateway "succeeded" → Odoo "paid"
  - Gateway "failed" → Odoo "failed"
  - Gateway "pending" → Odoo "pending"
- [ ] Enrollment trigger: Post-payment webhook to Website group `/api/v1/elearning/enrollment-trigger`
- [ ] Retry logic: If enrollment trigger fails, retry max 3x with exponential backoff
- [ ] Logging: All webhooks logged to audit trail with full payload + response
- [ ] Error handling: Failed webhook processing logged but not blocking

**Webhook Handler Implementation**:
```python
@route(['/payment/webhook/<gateway_code>'], type='http', auth='public', csrf=False, methods=['POST'])
def webhook_handler(self, gateway_code):
    # 1. Verify signature
    payload = request.get_json_data()
    signature = request.headers.get('X-Signature')
    
    gateway = payment_gateway.search([('gateway_code', '=', gateway_code)])
    if not gateway or not self._verify_signature(payload, signature, gateway.webhook_secret):
        return jsonify({'status': 'SIGNATURE_INVALID'}), 401
    
    # 2. Check idempotency
    reference = payload['reference']
    if payment_transaction.search([('reference', '=', reference), ('webhook_processed', '=', True)]):
        return jsonify({'status': 'ALREADY_PROCESSED'}), 200
    
    # 3. Update payment state
    tx = payment_transaction.search([('reference', '=', reference)])
    tx.update_from_webhook(payload)
    
    # 4. Trigger enrollment (async job)
    tx._trigger_enrollment_async()
    
    # 5. Log webhook
    self._log_webhook_event(gateway_code, payload, 'SUCCESS')
    
    return jsonify({'status': 'OK'}), 200
```

**Database Schema for Webhooks**:
```
payment.webhook_log:
  - id (primary)
  - gateway_id (FK to payment.gateway)
  - reference (varchar)
  - payload (json)
  - signature_verified (bool)
  - status (selection: 'RECEIVED', 'PROCESSED', 'FAILED', 'IGNORED')
  - processing_error (text, nullable)
  - idempotency_key (varchar, unique)
  - received_at (datetime)
  - processed_at (datetime, nullable)
  - retry_count (int, default 0)
  - next_retry_at (datetime, nullable)
```

#### User Story 4: Payment Failure & Retry

**Given**: Customer payment fails  
**When**: Gateway returns failure response  
**Then**: System handles failure gracefully

**Acceptance Criteria**:
- [ ] Failure detection: On receipt of failed webhook
- [ ] State update: payment_transaction.state = 'failed'
- [ ] Reason recording: failure_reason populated with gateway message
- [ ] Customer notification: Email sent to customer with failure reason + retry link
- [ ] Retry mechanism: Link to retry payment (re-initiate checkout, not duplicate charge)
- [ ] Limit: Max 3 retry attempts per payment
- [ ] Order preservation: sale.order remains in "draft" until payment succeeds
- [ ] Cleanup: Failed transactions older than 7 days archived

**Failure Scenarios Handled**:
- Card declined (insufficient funds)
- Card expired
- CVV mismatch
- Timeout / connection error
- Gateway maintenance
- 3D Secure authentication failed

**Boundary Clarification (Hybrid)**:
- Payment sukses tidak otomatis berarti peserta boleh ujian akhir pada course hybrid.
- Hak buka ujian tetap ditentukan Website berdasarkan status check-in attendance.

#### User Story 7: B2B Revenue Recognition Report

**Given**: HR wants to see which redeem codes were used  
**When**: HR runs reconciliation report  
**Then**: Report shows usage per contract

**Acceptance Criteria**:
- [ ] Report availability: Accounting → Reports → "B2B Redeem Code Reconciliation"
- [ ] Filters: By date range, contract, course
- [ ] Columns: Contract #, Course name, Codes allocated, Codes claimed, Codes unused, Expiry date, Revenue amount
- [ ] Export: CSV export functionality
- [ ] Drill-down: Click on contract to see individual student enrollments
- [ ] Auto-calculation: Unused codes value calculated (for contract closeout)

**Report Query**:
```
SELECT 
  contract.name,
  course.name,
  COUNT(CASE WHEN rc.isClaimed=false THEN 1 END) as unused,
  COUNT(CASE WHEN rc.isClaimed=true THEN 1 END) as used,
  SUM(course.price) as revenue,
  rc.expiry_date
FROM elearning.redeem_code rc
JOIN sale.order contract ON rc.contract_id = contract.id
JOIN elearning.course course ON rc.course_id = course.id
GROUP BY contract.id, course.id
```

## 5. Technical Specifications

### 5.1 Development Stack

| Component | Technology | Notes |
|---|---|---|
| **Payment SDK** | Gateway-specific SDKs (stripe-python, ipaymu-sdk) | Vendor libraries |
| **Webhook Framework** | Odoo routing + Werkzeug | Native Odoo webhooks |
| **Encryption** | cryptography.fernet | For API key storage |
| **HTTP Client** | requests library | For API calls to gateway |
| **Async Jobs** | Odoo ir.cron + ir.mail_server | For retries & notifications |
| **Logging** | Python logging + Odoo audit | Sensitive data masked |
| **Testing** | unittest + mock (responses library) | Mock API responses |

### 5.2 Security Considerations

- **PCI DSS Compliance**: No credit card data stored locally (all handled by payment gateway)
- **Webhook Signature**: Verify HMAC-SHA256 signature on every webhook
- **API Keys**: Encrypted in database + masked in logs
- **Rate Limiting**: Prevent brute force on payment API (max 10 requests/minute per IP)
- **Timeout**: Set reasonable timeouts on API calls (default 30s)
- **Error Messages**: Don't expose gateway response details to frontend (log internally only)
- **Audit Trail**: Log all payment transactions for compliance

### 5.3 Error Handling Strategy

| Error Type | Handling |
|---|---|
| **Connection timeout** | Retry 3x with exponential backoff; mark as "processing" |
| **Invalid amount** | Reject payment, notify customer |
| **Duplicate transaction** | Use idempotency key, return existing result |
| **Gateway down** | Return user-friendly message, preserve order state |
| **Webhook signature invalid** | Log security event, discard webhook |
| **Invoice generation failed** | Trigger alert, allow manual retry |

### 5.4 Performance Requirements

| Feature | Target | Measurement |
|---|---|---|
| Payment processing | < 2s | End-to-end (excluding gateway delay) |
| Webhook processing | < 500ms | Per webhook |
| Invoice generation | < 1s | Per invoice |
| Reconciliation report | < 5s | For 10K transactions |
| Concurrent transactions | 100+ | Per minute |
