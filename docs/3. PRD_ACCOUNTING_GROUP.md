## 1. Executive Summary

Accounting Group bertanggung jawab untuk **payment processing dan financial reconciliation** dalam platform Microcredential IMPC. 

> **Prinsip Odoo 19**: 
> - **Xendit** (`payment_xendit`) adalah modul **NATIVE** Odoo 19 Community yang sudah ada di official source — bukan third-party. Hanya perlu aktivasi dan konfigurasi.
> - **Midtrans** TIDAK ada di official Odoo 19 Community source. Harus install dari Odoo Apps Store (third-party, verifikasi kompatibilitas v19 terlebih dahulu).
> - Model invoice di Odoo 19 adalah `account.move` (bukan `account.invoice`). `account.invoice` sudah dihapus sejak Odoo 14.

**Deliverables** (Kustomasi Minor — Install + Config + Minor Code):
- Instalasi & konfigurasi modul payment provider Midtrans/Xendit dari Odoo Apps (cek kompatibilitas v19)
- Konfigurasi Indonesian Localization + PPN 11%
- Enrollment trigger ke Website Group setelah pembayaran dikonfirmasi
- Redeem code transaction reconciliation untuk B2B revenue tracking
- Invoice metadata enrichment (learning_mode per line item)

**Deliverables** (Standar — Native Odoo 19):
- Invoice generation (`account.move` native, bukan `account.invoice`)
- Payment state management
- Financial reporting
- Ledger entries & audit trail

**Catatan Penting**: Accounting Group TIDAK membangun payment gateway dari nol. Model `payment.provider` dan `payment.transaction` sudah ada di Odoo 19 secara native. **Xendit NATIVE** (modul `payment_xendit` ada di official Odoo 19 source). Model invoice di Odoo 19 adalah `account.move` (bukan `account.invoice` yang sudah dihapus sejak Odoo 14).

#### User Story 2: Konfigurasi Enrollment Trigger Post-Payment

**Given**: Pembayaran kursus B2C dikonfirmasi oleh payment provider  
**When**: `sale.order` dikonfirmasi via `_action_confirm()` setelah payment  
**Then**: Enrollment peserta ke course otomatis terjadi

> **KOREKSI ODOO 19 — ARSITEKTUR ENROLLMENT**: Di Odoo 19 dengan modul `website_sale_slides`, enrollment B2C terjadi secara NATIVE melalui `sale.order._action_confirm()`. Accounting Group TIDAK perlu memanggil Website API atau membuat Automated Action untuk enrollment B2C. Cukup:
> 1. Install modul `website_sale_slides`
> 2. Set `slide.channel.enroll = 'payment'` pada setiap course yang dijual
> Enrollment otomatis terjadi saat `sale.order` dikonfirmasi (yang terjadi setelah payment provider callback).

**Acceptance Criteria**:
- [ ] Install modul `website_sale_slides` (native Odoo 19) — ini yang menangani enrollment otomatis
- [ ] Set `slide.channel.enroll = 'payment'` untuk semua course yang dijual via eCommerce
- [ ] Verifikasi: Setelah `sale.order._action_confirm()`, record `slide.channel.partner` dibuat otomatis
- [ ] Logging: Native Odoo chatter di `sale.order` menampilkan log enrollment
- [ ] Jika diperlukan custom trigger (kasus non-eCommerce/non-standard): Gunakan Automated Action pada `account.move` saat `payment_state='paid'` dengan filter `move_type='out_invoice'`

> ⚠️ **PENTING**: Accounting Group TIDAK perlu membuat endpoint `/api/v1/elearning/enrollment-trigger` atau memanggil Website API untuk enrollment B2C. Ini adalah arsitektur yang salah untuk Odoo same-instance deployment.

**Jika diperlukan custom trigger** (untuk kasus non-standard):
```python
# Automated Action: Settings → Technical → Automation
# Model: account.move
# Trigger: When a record is updated
# Filter: payment_state = 'paid' AND move_type = 'out_invoice'
# Action: Execute Python Code

for invoice in records:
    course_lines = invoice.invoice_line_ids.filtered(
        lambda l: l.slide_channel_id
    )
    for line in course_lines:
        channel = line.slide_channel_id
        partner = invoice.partner_id
        # Check if already enrolled (idempotency)
        existing = env['slide.channel.partner'].search([
            ('channel_id', '=', channel.id),
            ('partner_id', '=', partner.id)
        ], limit=1)
        if not existing:
            channel._action_add_member(target_partner=partner)
```

#### User Story 3: Payment Failure Handling

**Given**: Pembayaran gagal (ditangani oleh modul Midtrans/Xendit native)  
**When**: `payment.transaction.state` berubah ke `'error'` atau `'cancel'`  
**Then**: Invoice tetap di status draft; sale.order tidak diproses

**Acceptance Criteria**:
- [ ] Tidak ada custom code diperlukan — modul payment provider mengelola failure state
- [ ] `sale.order` tetap di state `'sale'` (confirmed) tapi belum invoiced jika payment gagal
- [ ] `payment.transaction.state` saat payment berhasil = `'done'` (BUKAN `'paid'`)
- [ ] `payment.transaction.state` values Odoo 19: `'draft'`, `'pending'`, `'authorized'`, `'done'`, `'cancel'`, `'error'`
- [ ] Customer dapat retry payment melalui link yang disediakan modul payment provider
- [ ] Admin dapat melihat status payment di `Accounting → Payment Providers → Transactions`
- [ ] Log failure tersimpan di native `payment.transaction` record

> ⚠️ **KOREKSI ODOO 19**: `payment.transaction.state='paid'` TIDAK ADA. State yang benar untuk pembayaran sukses adalah `payment.transaction.state='done'`. Jangan bingungkan dengan `account.move.payment_state='paid'` yang BERBEDA dan VALID.

**Boundary Clarification (Hybrid)**:
- Payment sukses TIDAK otomatis membuka akses ujian akhir pada course hybrid
- Hak buka ujian ditentukan Website Group berdasarkan status check-in Events attendance

**Given**: Payment confirmed by payment gateway  
**When**: webhook processed and `payment_transaction.state='done'` (BUKAN `'paid'`)  
**Then**: Invoice auto-generated from sale.order

**Acceptance Criteria**:
- [ ] Invoice trigger: Scheduled action or direct trigger on payment confirmation
- [ ] Invoice source: Created from sale.order (course product)
- [ ] Line items: Course name, price, quantity=1, description
- [ ] Invoice type: Standard "Customer Invoice" (not refund, not credit note)
- [ ] Customer: Invoice customer matched to sale.order customer
- [ ] Amount: Invoice total matches payment amount
- [ ] Payment link: Invoice linked to payment_transaction record
- [ ] Invoice state: Invoice marked as "posted" (not draft)
- [ ] Notification: Invoice sent to customer email
- [ ] Ledger: Accounting entries automatically created (debit receivable, credit revenue)

**Database Schema**:
```
# account.invoice TIDAK ADA di Odoo 19. Model yang benar adalah account.move.
account.move (extended, move_type='out_invoice'):
  - payment_transaction_id (Many2one ke payment.transaction, nullable)
  - slide_channel_id (Many2one ke slide.channel, nullable)  # BUKAN course_id ke elearning.course
  - invoice_category (Selection: 'COURSE_B2C', 'COURSE_B2B', 'STANDARD', default 'STANDARD')
```

> ⚠️ **KOREKSI ODOO 19**: `account.invoice` TIDAK ADA. Gunakan `account.move` dengan `move_type='out_invoice'`. FK course harus referensi ke `slide.channel` (bukan `elearning.course`).

**Workflow**:
```
sale.order (draft) → Payment submitted → payment_transaction (pending)
↓
Payment confirmed → payment_transaction (done)  # state='done' BUKAN 'paid'
↓
B2C Enrollment: sale.order._action_confirm() via website_sale_slides (NATIVE, OTOMATIS)
↓
Invoice auto-generated → account.move (posted, move_type='out_invoice') → Sent to customer
```

### 3.3 Redeem Code Reconciliation (SKENARIO 1-2: B2B)

#### User Story 6: Redeem Code Transaction Tracking

**Given**: B2B customer uses redeem code to enroll student  
**When**: Code validated and student enrolled (via Website module)  
**Then**: Transaction recorded for reconciliation

**Acceptance Criteria**:
- [ ] Transaction record: entry in `elearning.redeem_code_transaction`
- [ ] Mapping: Links redeem_code_id, student_id, course_id, enrollment_id
- [ ] Value: Record course price as transaction amount (for reporting)
- [ ] Date: Timestamp of when student claimed code
- [ ] Status: Mark as 'CLAIMED', indicating course slot consumed
- [ ] No payment: Unlike B2C, no actual payment transaction (already paid via contract)
- [ ] Reporting: Transactions aggregatable for B2B revenue reporting

**Database Schema**:
```
# impc.redeem_code_transaction (prefix impc untuk model custom)
impc.redeem_code_transaction:
  - id (primary)
  - redeem_code_id (Many2one ke impc.redeem_code, index=True)
  - slide_channel_id (Many2one ke slide.channel)  # BUKAN course_id ke elearning.course
  - partner_id (Many2one ke res.partner)  # student
  - channel_partner_id (Many2one ke slide.channel.partner)  # enrollment
  - transaction_type (Selection: 'CLAIM')
  - amount (Float)  # Course price at time of claim
  - currency_id (Many2one ke res.currency, default IDR)
  - claimed_date (Datetime)
  - create_date (Datetime)  # gunakan Odoo native create_date
  
account.move (ledger entry, for B2B revenue recognition, extended):
  - redeem_code_transaction_id (Many2one ke impc.redeem_code_transaction, nullable)
  - description (Text, e.g., "Course enrollment via redeem: ABC123")
```

> ⚠️ **KOREKSI ODOO 19**: `elearning.redeem_code_transaction` diganti dengan `impc.redeem_code_transaction`. FK ke course menggunakan `slide_channel_id` (ke `slide.channel`). FK ke enrollment menggunakan `channel_partner_id` (ke `slide.channel.partner`).

#### User Story 7: B2B Revenue Recognition Report

**Given**: HR wants to see which redeem codes were used  
**When**: HR runs reconciliation report  
**Then**: Report shows usage per contract

**Acceptance Criteria**:
- [ ] Report availability: Accounting → Reports → "B2B Redeem Code Reconciliation"
- [ ] Filters: By date range, contract, course
- [ ] Columns: Contract #, Course name, Codes allocated, Codes claimed, Codes unused, Expiry date, Revenue amount
- [ ] Export: CSV export functionality
- [ ] Drill-down: Click on contract to see individual student enrollments
- [ ] Auto-calculation: Unused codes value calculated (for contract closeout)

**Report Query**:
```sql
-- Gunakan nama tabel Odoo 19 yang benar
-- impc_redeem_code = tabel DB dari model impc.redeem_code
-- slide_channel = tabel DB dari model slide.channel (eLearning course)
SELECT 
  so.name as contract_name,
  sc.name as course_name,
  COUNT(CASE WHEN rc.is_claimed = false THEN 1 END) as unused,
  COUNT(CASE WHEN rc.is_claimed = true THEN 1 END) as used,
  SUM(sc.website_price) as revenue,
  rc.expiry_date
FROM impc_redeem_code rc
JOIN sale_order so ON rc.contract_id = so.id
JOIN slide_channel sc ON rc.channel_id = sc.id
GROUP BY so.id, sc.id
```

> ⚠️ **KOREKSI**: Tabel DB untuk eLearning course di Odoo 19 adalah `slide_channel` (dari model `slide.channel`), BUKAN `elearning_course`. Field is_claimed menggunakan snake_case (sesuai konvensi Odoo ORM).

## 5. Technical Specifications

### 5.1 Development Stack

| Component | Technology | Notes |
|---|---|---|
| **Payment Provider** | Modul Midtrans/Xendit dari Odoo Apps Store | Bukan custom dev — install + config |
| **Enrollment Trigger** | Odoo Automated Action + requests library | Custom: hanya bagian ini |
| **Invoice Extension** | Odoo ORM (inheritance `account.move.line`) | Minor: tambah field `course_id`, `learning_mode` |
| **Reconciliation Report** | Odoo Pivot View + custom SQL query | Untuk B2B redeem code tracking |
| **Async Retry** | Odoo `ir.cron` | Native scheduler untuk retry enrollment trigger |
| **Logging** | Native Odoo chatter (`mail.message`) | Log enrollment trigger per invoice |
| **Testing** | Odoo sandbox + staging environment | Test dengan akun sandbox Midtrans/Xendit |

### 5.2 Security Considerations

- **PCI DSS Compliance**: Tidak ada data kartu kredit tersimpan di Odoo \u2014 semua dihandle oleh Midtrans/Xendit
- **Webhook Signature**: Diverifikasi otomatis oleh modul third-party (tidak perlu implementasi manual)
- **API Keys**: Disimpan di field native `payment.provider` Odoo (encrypted oleh framework)
- **Enrollment Trigger API Key**: Simpan di `ir.config_parameter` (Settings \u2192 Technical \u2192 Parameters \u2192 System Parameters), bukan di kode
- **Audit Trail**: Log enrollment trigger di native Odoo chatter per invoice

### 5.3 Error Handling Strategy

| Error Type | Handling |
|---|---|
| **Connection timeout** | Retry 3x with exponential backoff; mark as "processing" |
| **Invalid amount** | Reject payment, notify customer |
| **Duplicate transaction** | Use idempotency key, return existing result |
| **Gateway down** | Return user-friendly message, preserve order state |
| **Webhook signature invalid** | Log security event, discard webhook |
| **Invoice generation failed** | Trigger alert, allow manual retry |

### 5.4 Performance Requirements

| Feature | Target | Measurement |
|---|---|---|
| Payment processing | < 2s | End-to-end (excluding gateway delay) |
| Webhook processing | < 500ms | Per webhook |
| Invoice generation | < 1s | Per invoice |
| Reconciliation report | < 5s | For 10K transactions |
| Concurrent transactions | 100+ | Per minute |
